## 2. 設計思想

### 基本方針: Arduino = 最小限のドライバ / Raspberry Pi = すべての知能

| 方針 | 理由 |
|------|------|
| Arduino は「読み取り」と「ON/OFF」だけ | 機能改善のたびにArduinoを書き換えたくない |
| 判定ロジック・閾値はすべてラズパイ側 | Pythonで柔軟に変更可能 |
| Arduino のファームウェアは汎用コマンド型 | 一度書き込めば基本的に変更不要 |

---

## 3. Arduino の機能 (ドライバ層)

### 役割: シリアルコマンドを受け取り、センサー値を返す or アクチュエータを操作する

Arduino は **コマンド・レスポンス型** のシンプルなプロトコルのみ実装する。
自律的な判断は一切行わない。

### 3.1 必須機能 (Must)

| # | 機能 | コマンド例 | レスポンス例 | 説明 |
|---|------|-----------|-------------|------|
| A1 | 土壌湿度読み取り | `READ_SOIL` | `SOIL:512,480` | 全センサーの生値をカンマ区切りで返す |
| A2 | 水位読み取り | `READ_WATER` | `WATER:1` | フロートスイッチの状態 (1=正常 / 0=水不足) |
| A3 | ポンプON | `PUMP_ON` | `OK:PUMP_ON` | ラッチリレーをON (給水開始) |
| A4 | ポンプOFF | `PUMP_OFF` | `OK:PUMP_OFF` | ラッチリレーをOFF (給水停止) |
| A5 | 全センサー一括読み取り | `READ_ALL` | `SOIL:512,480;WATER:1` | デバッグ・定期取得用 |
| A6 | ヘルスチェック | `PING` | `PONG` | 通信確認用 |

### 3.2 実装推奨機能 (Should)

| # | 機能 | コマンド例 | レスポンス例 | 説明 |
|---|------|-----------|-------------|------|
| A7 | ポンプ状態確認 | `STATUS_PUMP` | `PUMP:ON` or `PUMP:OFF` | 現在のリレー状態を返す |
| A8 | 温湿度読み取り | `READ_DHT` | `DHT:25.3,60.2` | DHT22から温度・湿度 (動画でも使用) |
| A9 | 安全タイムアウト | (内部処理) | - | ポンプON後、一定時間(例:60秒)で自動OFF |
| A10 | エラー応答 | - | `ERR:UNKNOWN_CMD` | 不明コマンドに対するエラー返却 |

### 3.3 あると便利な機能 (Nice to Have)

| # | 機能 | 説明 |
|---|------|------|
| A11 | ファームウェアバージョン応答 (`VERSION`) | ラズパイ側で互換性チェックに使える |
| A12 | センサーキャリブレーション値の保持 | EEPROMに保存し、`SET_CAL` / `GET_CAL` で操作 |

### 3.4 Arduino 安全設計 (重要)

```
⚠️ Arduino側に最低限の安全弁を持たせる (ラズパイがフリーズしても被害を防ぐ)
```

| 安全機能 | 内容 |
|---------|------|
| **ポンプ自動OFF タイムアウト** | `PUMP_ON` 後、ラズパイからの応答がなくても **最大60秒で自動OFF** |
| **水位不足時のポンプ禁止** | `PUMP_ON` 受信時に水位センサーが 0 なら `ERR:NO_WATER` を返しポンプ起動しない |
| **ウォッチドッグ** | シリアル通信が一定時間途絶えた場合、ポンプを安全にOFF |

> これらは「Arduino ファームウェアの変更不要」原則の **例外**。
> 安全機能だけは Arduino 側にハードコードする。

## 4. Raspberry Pi の機能 (制御・ロジック層)

### 役割: すべての判断・スケジューリング・外部連携を担当

### 4.1 必須機能 (Must)

| # | 機能 | 説明 |
|---|------|------|
| R1 | **Arduinoシリアル通信ドライバ** | コマンド送受信、タイムアウト処理、リトライ |
| R2 | **定期センサー取得** | cron or スケジューラで定期的に `READ_ALL` を実行し記録 |
| R3 | **給水判定ロジック** | 土壌湿度が閾値以下 → 水位確認 → ポンプON → 一定時間後OFF |
| R4 | **給水スケジューラ** | 設定した時刻(例: 毎朝7:00)に判定を実行 |
| R5 | **Google Spreadsheet 読み取り** | スマホから書き込まれた設定値 (閾値、スケジュール等) を取得 |
| R6 | **Google Spreadsheet 書き込み** | センサーデータ・給水履歴をシートに記録 |
| R7 | **設定管理** | 閾値・スケジュール・通知設定をファイル or シートで管理 |

### 4.2 実装推奨機能 (Should)

| # | 機能 | 説明 |
|---|------|------|
| R8 | **Discord / LINE 通知** | 給水実行・水不足アラート・日次レポートを送信 |
| R9 | **データグラフ生成** | matplotlib で湿度推移グラフを生成し通知に添付 |
| R10 | **USBカメラ撮影** | 定期的に植物を撮影してDiscordに送信 (成長記録) |
| R11 | **異常検知・アラート** | センサー値異常、Arduino応答なし、水位不足を検知して即通知 |
| R12 | **ログ記録** | すべての操作・センサー値をローカルファイルにも保存 |
| R13 | **手動給水コマンド** | Spreadsheet から「今すぐ給水」を指示可能にする |

### 4.3 あると便利な機能 (Nice to Have)

| # | 機能 | 説明 |
|---|------|------|
| R14 | **Webダッシュボード** | ラズパイ上でFlask等の簡易Webサーバーを立て、状態をブラウザで確認 |
| R15 | **天気API連携** | 雨予報なら給水スキップ (屋外の場合) |
| R16 | **学習型閾値調整** | 過去データから最適な給水閾値を自動調整 |
| R17 | **OTA的な設定反映** | Spreadsheetの設定変更を即座にポーリングして反映 |

## 6. 通信プロトコル詳細

### 6.1 シリアル通信仕様

| 項目 | 値 |
|------|----|
| ボーレート | 9600 bps |
| データビット | 8 |
| ストップビット | 1 |
| パリティ | なし |
| 改行コード | `\n` (LF) |
| タイムアウト | 2秒 |
| リトライ回数 | 3回 |

## 7. 給水判定ロジック (ラズパイ側)

```python
# 疑似コード
def watering_check():
    # 1. センサー値取得
    soil_values = arduino.send("READ_SOIL")   # → [320, 410]
    water_level = arduino.send("READ_WATER")  # → 1

    # 2. 設定値取得 (Spreadsheet or ローカル設定)
    config = load_config()
    threshold = config["soil_threshold"]       # → 400
    pump_duration = config["pump_duration"]    # → 10秒

    # 3. 給水判定
    avg_moisture = average(soil_values)
    if avg_moisture >= threshold:
        log("土壌湿度十分。給水不要。")
        return

    # 4. 水位確認
    if water_level == 0:
        alert("⚠️ 給水タンクの水が不足しています！")
        return

    # 5. 給水実行
    arduino.send("PUMP_ON")
    sleep(pump_duration)
    arduino.send("PUMP_OFF")

    # 6. 給水後の確認
    sleep(30)  # 水が土壌に浸透するのを待つ
    soil_after = arduino.send("READ_SOIL")

    # 7. 結果記録・通知
    log_to_spreadsheet(soil_values, soil_after, pump_duration)
    notify(f"給水完了: {avg_moisture} → {average(soil_after)}")
```

## 11. 注意点・設計判断メモ

### なぜ Arduino をドライバに徹させるか
1. **ファーム更新の手間**: Arduino書き換えには物理接続 or OTA設定が必要
2. **デバッグの容易さ**: Python側ならprint文で即座にデバッグ可能
3. **柔軟性**: 閾値変更・ロジック追加がPythonコードの編集だけで完結
4. **テスト容易性**: Arduinoモックを使ったユニットテストが可能

### 安全設計の考え方
- **最悪ケース = ポンプが止まらない → 水浸し**
- Arduino側のタイムアウト (A9) は **必ず** 実装する
- ラズパイがフリーズしても Arduino 単体でポンプを止められること
- 水位不足時のポンプ禁止 (Arduino側) は二重安全として重要

### Spreadsheet の限界
- APIの呼び出し制限 (100リクエスト/100秒/ユーザー)
- リアルタイム性はない (ポーリング間隔に依存)
- 将来的にはMQTT + アプリに置き換える価値あり
