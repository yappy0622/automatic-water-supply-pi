## 2. 設計思想

### 基本方針: Arduino = 最小限のドライバ / Raspberry Pi = すべての知能

| 方針 | 理由 |
|------|------|
| Arduino は「読み取り」と「ON/OFF」だけ | 機能改善のたびにArduinoを書き換えたくない |
| 判定ロジック・閾値はすべてラズパイ側 | Pythonで柔軟に変更可能 |
| Arduino のファームウェアは汎用コマンド型 | 一度書き込めば基本的に変更不要 |

### 省電力設計

| パラメータ | デフォルト値 | 説明 |
|-----------|------------|------|
| `main_loop_sleep_sec` | 30秒 | メインループのスリープ間隔 |
| `sensor_interval_min` | 30分 | センサー定期取得の間隔 |
| `sheets_poll_interval_min` | 30分 | Spreadsheet ポーリング間隔 |
| `graph_interval_hours` | 6時間 | グラフ自動生成間隔 |

- **ポイント**: センサー記録と Sheets ポーリングを同じ 30 分周期に統合
- Sheets API の呼び出し回数を 1 日 48 回に抑制 (制限の 2.3%)
- メインループは 30 秒スリープで CPU 使用率を最小化

## 3. Arduino の機能 (ドライバ層)

### 役割: シリアルコマンドを受け取り、センサー値を返す or アクチュエータを操作する

Arduino は **コマンド・レスポンス型** のシンプルなプロトコルのみ実装する。
自律的な判断は一切行わない。

### 3.1 コマンド一覧 (v1.1.0)

| # | コマンド | レスポンス | 説明 |
|---|---------|-----------|------|
| A1 | `READ_SOIL` | `SOIL:512,480` | 全センサーの生値をカンマ区切りで返す |
| A2 | `READ_WATER` | `WATER:1` | フロートスイッチの状態 (1=正常 / 0=水不足) |
| A3 | `READ_LIGHT` | `LIGHT:350.0` | 照度 (lux) |
| A4 | `READ_DHT` | `DHT:25.3,60.2` | 温度・湿度 |
| A5 | `READ_ALL` | `SOIL:..;WATER:..;DHT:..;LIGHT:..;PUMP:..` | 全データ一括 |
| A6 | `PUMP_ON` | `OK:PUMP_ON` / `ERR:NO_WATER` | ポンプ起動 |
| A7 | `PUMP_OFF` | `OK:PUMP_OFF` | ポンプ停止 |
| A8 | `STATUS_PUMP` | `PUMP:ON` / `PUMP:OFF` | ポンプ状態 |
| A9 | `PING` | `PONG` | ヘルスチェック |
| A10 | `VERSION` | `VERSION:WateringDriver,1.1.0` | FW バージョン |

### 3.2 安全設計 (重要)

```
⚠️ Arduino側に最低限の安全弁を持たせる (ラズパイがフリーズしても被害を防ぐ)
```

| 安全機能 | 内容 |
|---------|------|
| **ポンプ自動OFF タイムアウト** | `PUMP_ON` 後、最大60秒で自動OFF |
| **水位不足時のポンプ禁止** | 水位0のとき PUMP_ON を拒否 (`ERR:NO_WATER`) |
| **給水中水位監視** | 給水中に水がなくなったら即停止 |
| **ウォッチドッグ** | シリアル通信が5分途絶えた場合、ポンプ停止 |

> これらは「Arduino ファームウェアの変更不要」原則の **例外**。
> 安全機能だけは Arduino 側にハードコードする。

| 項目 | 値 |

| データビット | 8 |
| ストップビット | 1 |
| パリティ | なし |
| 改行コード | `\n` (LF) |

## 10. 注意点・設計判断メモ

### なぜ Arduino をドライバに徹させるか
1. **ファーム更新の手間**: Arduino書き換えには物理接続 or OTA設定が必要
2. **デバッグの容易さ**: Python側ならprint文で即座にデバッグ可能
3. **柔軟性**: 閾値変更・ロジック追加がPythonコードの編集だけで完結
4. **テスト容易性**: Arduinoモックを使ったユニットテストが可能

### 安全設計の考え方
- **最悪ケース = ポンプが止まらない → 水浸し**
- Arduino側のタイムアウトは **必ず** 実装する
- ラズパイがフリーズしても Arduino 単体でポンプを止められること
- 水位不足時のポンプ禁止 (Arduino側) は二重安全として重要

### Spreadsheet の限界
- APIの呼び出し制限 (100リクエスト/100秒/ユーザー)
- リアルタイム性はない (30分ポーリング間隔)
- 将来的にはMQTT + アプリに置き換える価値あり

### 省電力の考え方
- スマホからの設定変更は頻繁ではない → 30分ポーリングで十分
- センサー記録とポーリングを同周期に統合 → API 呼び出し削減
- メインループ 30秒スリープ → CPU 使用率最小化
